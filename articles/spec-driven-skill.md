## はじめに

1年ほど前から、自分用のプランモード＆実装モード用のスキルを作って運用していますが、個人的にかなり良い感じで実装ができています。

元々作成しようと思ったきっかけは、Kiro の Spec モードを使用した際に、実装計画とタスクの md ファイルを作成することに感銘を受け、Claude Code でもこれに似たファイルを作成して実装を進めたいと考えたからです。

## プランモードの限界

Claude Code の標準のプランモードは、基本的に Ask ツールでのやり取りの結果が md ファイル1つに書き出されます。ただ、運用していく中で以下のような不便さを感じるようになりました。

- 探索結果やユーザーへの質問内容などを、独立したファイルとして残せない
- 出力先が md に固定されるため、技術的な解説を HTML で作らせるといった応用がしづらい
- 「計画中は実装させたくない」という運用ルールを、仕組みとして担保できない

こうした不満を踏まえて、実装計画・タスクに加えてテストケースや技術解説なども生成でき、かつ自分なりのフォーマットを強制できるスキルを作ることにしました。狙いは、実装時の認知負荷を下げること・テストケースの抜け漏れを減らすこと・実装計画を徹底的に詰めることの3つです。

## 作ったスキルの全体像

大きくは、以下の2つのスキルを組み合わせて使っています。

- **プランモードスキル**: 探索・要件整理・実装計画とタスクへの落とし込みまでを行う
- **実装モードスキル**: プランモードで作成した成果物をもとに実装を進める

プランモードで作成したファイルは、計画ごとにフォルダにまとめ、フォルダ名にインデックス（連番）を付与します。実装モードはこのインデックスを引数で受け取り、省略時は最新のものを対象とします。

出力されるフォルダ構成のイメージは以下です。

```
.plugin-workspace/.specs/001-feature-name/
├── PLANNING            # フェーズマーカー（存在する間は計画中）
├── requirements.md     # 要件
├── design.md           # 設計
├── tasks.md            # タスク一覧（□ / ■ で進捗管理）
└── notes/              # 探索結果や調査メモ
```

この後の章では、このスキルの中で工夫している2つのポイント、「出力の分割設計」と「Hooks によるフェーズのガード」について書いていきます。

## 工夫①: 出力を分割する設計

標準のプランモードのように1枚の md にすべてを詰め込むのではなく、フェーズや用途ごとにファイルを分けて出力する構成にしました。要件・設計・タスク・調査メモをそれぞれ別ファイルに切り出しておく、というシンプルな方針です。

分割しておくと、次のような嬉しさがあります。

- 特定のフェーズだけあとから差し替えたり、HTML など別形式の成果物を追加したりしやすい
- レビュー時に「今読みたい範囲」だけを開ける
- `PLANNING` ファイルの有無をフェーズマーカーとして使えるので、後述するガード処理や自動アーカイブと連動させやすい

要は、md 1枚に詰め込んだ状態だと難しかった「フェーズ単位での取り回し」を、ファイル分割によって成立させている、というのがポイントです。

## 工夫②: Hooksによるガード

もう一つの軸が、プランモード中はソースコードなどのファイルを勝手に書き換えられないようにする、というガードの仕組みです。ここは Claude Code の Hooks を使って実現しています。

Claude Code の Hooks では、現在のセッション ID を含む情報が JSON で渡ってくるので、これを利用して「このセッションは今、計画モード中である」という状態をファイルで表現します。

具体的には、スキル起動時にセッション ID と同名のガードファイルを生成します。

```
.plugin-workspace/.specs/.guard/{SESSION_ID}
```

このファイルが存在している間は、`PreToolUse` フックで Write / Edit / Bash をブロックする、という単純な仕組みです。

### フック登録

`hooks.json` で、`Write` / `Edit` / `MultiEdit` / `Bash` に対して `PreToolUse` フックを登録しています。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          { "type": "command", "command": "bash \"${CLAUDE_PLUGIN_ROOT}/hooks/guard-planning-writes.sh\"" }
        ]
      },
      {
        "matcher": "Edit|MultiEdit",
        "hooks": [
          { "type": "command", "command": "bash \"${CLAUDE_PLUGIN_ROOT}/hooks/guard-planning-writes.sh\"" }
        ]
      },
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "bash \"${CLAUDE_PLUGIN_ROOT}/hooks/guard-planning-writes.sh\"" }
        ]
      }
    ]
  }
}
```

### ガードスクリプトの中身

シェルスクリプト側では、標準入力から渡ってくる JSON からセッション ID とツール名を取り出し、対応するガードファイルの有無で挙動を切り替えます。`exit 2` を返すとツール実行がブロックされ、標準エラー出力の内容がモデルへのフィードバックとして渡ります。

```bash
INPUT=$(cat)
SESSION_ID=$(echo "$INPUT" | jq -r '.session_id // empty')
TOOL=$(echo "$INPUT" | jq -r '.tool_name // empty')

GUARD_FILE=".plugin-workspace/.specs/.guard/${SESSION_ID}"

# ガードファイルが無ければ素通り
[ ! -f "$GUARD_FILE" ] && exit 0

# Write / Edit: .specs/ 以外への書き込みをブロック
if [ "$TOOL" = "Write" ] || [ "$TOOL" = "Edit" ] || [ "$TOOL" = "MultiEdit" ]; then
  FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')
  echo "$FILE" | grep -qE '(^|/)\.plugin-workspace/\.specs/' && exit 0

  echo "【計画モード】${FILE} への書き込みはブロックされました。" >&2
  exit 2
fi
```

Bash ツールについても同様に、リダイレクト（`>` / `>>`）や `tee` / `sed -i` / `cp` / `mv` / `rm` / `mkdir` といった書き込み系コマンドをパターンマッチで拾ってブロックしています。`.plugin-workspace/.specs/` 配下への操作だけは例外的に許可することで、計画ファイルの生成・更新はそのまま行えるようにしています。

### ガード解除

ガードは、ユーザーが手動でガードファイルを削除することで解除されます。モデル側から `rm` でガードファイルを消そうとした場合も、Bash フック側で検出してブロックするようにしているので、意図せず自動的に実装フェーズへ移行してしまうことを防げます。

## まとめ

このスキルの本質は、自分なりの出力形式で計画を立てられる点と、そのフェーズを Hooks で強制できる点です。プランモードそのものを AI に任せるのではなく、「どのファイルにどんな粒度で出力するか」「いつ実装を許可するか」という運用ルールを、自分の手元で設計できるのが気に入っています。
